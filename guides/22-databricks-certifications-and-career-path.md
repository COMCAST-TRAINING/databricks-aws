## 1. Overview of the Databricks Certification Programme

### 1.1 Why Get Databricks Certified?

Databricks certifications validate practical, hands-on proficiency with the Databricks Lakehouse Platform and Apache Spark. They signal to employers that you can build, optimise, and operate real production data systems — not just understand concepts.

**Benefits of certification:**

| Benefit                       | Detail                                                                                         |
| ----------------------------- | ---------------------------------------------------------------------------------------------- |
| **Market differentiation**    | Databricks skills are among the highest-paying in data engineering; a badge stands out on a CV |
| **Structured knowledge**      | Exam preparation forces systematic coverage of topics you may have learned piecemeal           |
| **Employer confidence**       | Hiring managers use certifications as a first filter for shortlisting candidates               |
| **Salary uplift**             | Certified Databricks engineers earn 15–30% more than non-certified peers in comparable roles   |
| **Consulting / partner work** | Many Databricks partner organisations require certified engineers for client engagements       |
| **Personal credibility**      | A publicly verifiable digital badge (via Credly) that you can share on LinkedIn                |

### 1.3 Exam Format, Delivery, and Policies

**Delivery format:**

- All exams are delivered online via **Webassessor** (Kryterion), proctored remotely.
- You can take the exam from your home or office — no testing centre required.
- A webcam, microphone, and a stable internet connection are required.

**Question format:**

- All questions are **multiple choice** (single correct answer) or **multiple select** (choose all that apply).
- No free-text, no practical lab component in the current exam format.
- Questions are based on **PySpark (Python)** for Data Engineering and Spark exams; SQL variants are tested in Data Analyst and Data Engineer exams.

**Exam policies:**

- **Closed book** — no notes, browser tabs, or documentation permitted during the exam.
- **No backtracking restriction** — you may flag questions and return before submitting.
- **Retake policy** — you may retake a failed exam after a **14-day waiting period**. A second failure requires a **60-day wait**.
- **No expiry restriction** on most certifications (Databricks does not currently publish a fixed renewal cycle, but certifications may be versioned — always check the Databricks website for current policy).

**Pricing (approximate, subject to change):**

- Associate-level: **USD 200** per attempt
- Professional-level: **USD 200** per attempt
- Retakes are separately charged at the same price.

> **Tip:** Databricks frequently runs promotions through their Learning platform and partner organisations. Check `databricks.com/learn/certification` for current pricing and voucher codes before purchasing.

---

### 1.4 Maintaining and Renewing Certifications

Databricks currently does not mandate a fixed renewal cycle the way AWS or Google Cloud do. However:

- When Databricks releases a **new exam version** (e.g. v3.0 replacing v2.0), older versions are retired on a 6–12 month migration schedule.
- You are expected to pass the updated version to hold a current certification.
- Check your **Credly profile** and the **Databricks certification page** periodically for retirement notices.

**Best practice:** Re-sit your certification whenever a new major version is released (typically every 1–2 years) to stay current with new features like Unity Catalog, Liquid Clustering, and Generative AI capabilities.

### 2.3 Key Topics to Master

**1. Spark DataFrame transformations vs actions:**

```python
# Transformations (lazy — build the DAG, don't execute)
df.select("col1", "col2")
df.filter(col("age") > 25)
df.groupBy("region").agg(avg("revenue"))
df.join(other_df, on="id", how="inner")
df.withColumn("new_col", col("a") + col("b"))

# Actions (trigger execution)
df.count()
df.show()
df.collect()
df.write.format("delta").save("/path")
df.first(), df.take(10)
```

**2. Understanding the DAG — stages and shuffles:**

```
DAG (Directed Acyclic Graph) for a groupBy + join query:

Stage 1 (no shuffle):
  Scan table_A → filter → project → [shuffle write]

Stage 2 (requires shuffle from Stage 1):
  [shuffle read] → groupBy aggregate → [shuffle write]

Stage 3 (requires shuffle from Stage 2):
  [shuffle read] → join → project → write output

Key rule: a new STAGE is created whenever a SHUFFLE is required.
```

**3. Key DataFrame operations:**

```python
from pyspark.sql import functions as F
from pyspark.sql.window import Window

# Window function (common exam question)
w = Window.partitionBy("department").orderBy(F.desc("salary"))
df.withColumn("rank", F.rank().over(w))

# Aggregation with multiple metrics
df.groupBy("region", "product") \
  .agg(
      F.sum("revenue").alias("total_revenue"),
      F.countDistinct("customer_id").alias("unique_customers"),
      F.avg("order_value").alias("avg_order"),
  )

# Join types
df1.join(df2, on="id", how="inner")   # only matching rows
df1.join(df2, on="id", how="left")    # all left rows + matching right
df1.join(df2, on="id", how="outer")   # all rows from both sides
df1.join(df2, on="id", how="anti")    # left rows with NO match in right
df1.join(df2, on="id", how="semi")    # left rows that have a match in right (no right cols)
```

**4. Reading and writing data sources:**

```python
# Reading with explicit schema (exam loves this pattern)
from pyspark.sql.types import StructType, StructField, StringType, IntegerType

schema = StructType([
    StructField("id",     IntegerType(), nullable=False),
    StructField("name",   StringType(),  nullable=True),
    StructField("salary", IntegerType(), nullable=True),
])

df = spark.read \
     .format("csv") \
     .option("header", "true") \
     .schema(schema) \
     .load("s3://bucket/employees/*.csv")

# Writing with partitioning
df.write \
  .format("delta") \
  .mode("overwrite") \
  .partitionBy("region") \
  .save("/mnt/data/employees_delta")
```

**5. Spark SQL and temp views:**

```python
# Register a DataFrame as a temp view (session-scoped)
df.createOrReplaceTempView("sales")

# Query using spark.sql()
result = spark.sql("""
    SELECT   region,
             SUM(revenue)                          AS total_revenue,
             RANK() OVER (ORDER BY SUM(revenue) DESC) AS revenue_rank
    FROM     sales
    GROUP BY region
""")

# Global temp view (cross-session, spark-scoped)
df.createOrReplaceGlobalTempView("global_sales")
spark.sql("SELECT * FROM global_temp.global_sales")
```

### 2.4 Sample Questions and Worked Answers

**Q1:** A DataFrame `df` has columns `[id, name, dept, salary]`. Which of the following returns only rows where `salary > 50000` AND `dept = 'Engineering'`?

- A. `df.filter(col("salary") > 50000).filter(col("dept") == "Engineering")`
- B. `df.filter((col("salary") > 50000) & (col("dept") == "Engineering"))`
- C. `df.where("salary > 50000 AND dept = 'Engineering'")`
- D. All of the above

**Answer: D** — All three are equivalent. `filter()` and `where()` are aliases. Chaining filters and using `&` for AND are both valid.

---

**Q2:** What is the output mode for a structured streaming query that outputs only **new rows** added since the last trigger?

- A. `complete`
- B. `append`
- C. `update`
- D. `incremental`

**Answer: B** — `append` mode emits only rows added in the current micro-batch. `complete` rewrites the entire result table every trigger. `update` emits only rows that were updated. `incremental` is not a valid mode.

---

**Q3:** Which action forces Spark to evaluate the full lineage of a DataFrame and return results to the driver?

- A. `df.filter(...)`
- B. `df.repartition(10)`
- C. `df.collect()`
- D. `df.select(...)`

**Answer: C** — `collect()` is an action that triggers execution and returns all rows to the driver. The others are transformations (lazy).

### 3.3 Key Topics to Master

**1. Auto Loader — the primary ingestion mechanism:**

```python
# Incremental ingestion with Auto Loader — understand every option
df = (
    spark.readStream
    .format("cloudFiles")
    .option("cloudFiles.format", "json")
    .option("cloudFiles.schemaLocation", "/checkpoints/schema/")
    .option("cloudFiles.inferColumnTypes", "true")
    .option("cloudFiles.includeExistingFiles", "true")   # process historical files too
    .load("s3://raw-data/events/")
)

(
    df.writeStream
    .format("delta")
    .option("checkpointLocation", "/checkpoints/events/")
    .option("mergeSchema", "true")
    .trigger(availableNow=True)   # process all pending files, then stop
    .toTable("bronze.raw_events")
)
```

**Key Auto Loader exam facts:**

- Uses `cloudFiles` format — not `csv`, `json`, etc. directly
- `schemaLocation` stores inferred schema; **required** for Auto Loader
- `availableNow=True` trigger — processes all files then stops (batch-like behaviour)
- `once=True` is deprecated; use `availableNow=True` instead
- Supports **directory listing** (default) and **file notifications** (SNS/SQS on AWS) modes

**2. MERGE INTO — upsert pattern:**

```python
from delta.tables import DeltaTable

target = DeltaTable.forName(spark, "silver.customers")

(
    target.alias("t")
    .merge(
        source=updates_df.alias("s"),
        condition="t.customer_id = s.customer_id"
    )
    .whenMatchedUpdate(set={
        "name":           "s.name",
        "email":          "s.email",
        "updated_at":     "s.updated_at",
    })
    .whenNotMatchedInsertAll()
    .whenNotMatchedBySourceDelete()   # delete rows no longer in source (SCD Type 1 full sync)
    .execute()
)
```

**3. Delta Live Tables (DLT):**

```python
import dlt
from pyspark.sql import functions as F

# Bronze — raw ingestion as a streaming table
@dlt.table(
    name="raw_orders",
    comment="Raw orders ingested from S3 via Auto Loader",
    table_properties={"quality": "bronze"}
)
def raw_orders():
    return (
        spark.readStream
        .format("cloudFiles")
        .option("cloudFiles.format", "json")
        .option("cloudFiles.schemaLocation", "/dlt/schema/orders")
        .load("s3://raw/orders/")
    )

# Silver — with DLT expectations for data quality
@dlt.table(name="cleaned_orders", comment="Validated orders")
@dlt.expect("valid_amount",     "order_amount > 0")
@dlt.expect_or_drop("has_customer", "customer_id IS NOT NULL")
@dlt.expect_or_fail("no_future_dates", "order_date <= current_date()")
def cleaned_orders():
    return (
        dlt.read_stream("raw_orders")
        .withColumn("order_date", F.to_date("order_date_str"))
        .drop("order_date_str")
    )
```

**DLT expectation actions — exam table:**

| Decorator             | On Violation                        |
| --------------------- | ----------------------------------- |
| `@dlt.expect`         | Record the violation; keep the row  |
| `@dlt.expect_or_drop` | Drop violating rows from the output |
| `@dlt.expect_or_fail` | Fail the entire pipeline update     |

### 3.4 Sample Questions and Worked Answers

**Q1:** An Auto Loader stream is reading JSON files from S3. After a schema change in the source (a new column is added), the stream should automatically incorporate the new column without failing. Which option enables this?

- A. `option("cloudFiles.schemaEvolutionMode", "addNewColumns")`
- B. `option("mergeSchema", "true")` on the write side
- C. Both A and B together are required
- D. Auto Loader handles schema evolution automatically with no options needed

**Answer: C** — You must set `schemaEvolutionMode` to `addNewColumns` on the read side AND `mergeSchema` to `true` on the write side. One without the other is insufficient.

---

**Q2:** A DLT table is decorated with `@dlt.expect_or_drop("valid_amount", "order_amount > 0")`. What happens to records where `order_amount = -5`?

- A. The pipeline fails immediately
- B. The record is written to a quarantine table
- C. The record is silently dropped from the output table
- D. The record is written with a warning flag column added

**Answer: C** — `expect_or_drop` silently drops rows violating the expectation. The violation is tracked in DLT event logs but the records do not appear in the output table.

---

**Q3:** A Workflow has tasks A → B → C in sequence. Task B fails. After fixing the bug in B, which Workflow feature allows re-running from Task B without re-running Task A?

- A. Re-run all failed tasks
- B. Repair Run
- C. Clone run
- D. Task retry policy

**Answer: B** — **Repair Run** lets you re-run from a specific failed task, reusing the results of already-succeeded tasks. This avoids re-running expensive upstream tasks unnecessarily.

### 4.3 Key Topics to Master

**1. Stateful Streaming and Watermarking:**

```python
from pyspark.sql import functions as F

# Watermarking — handle late-arriving data
events_with_watermark = (
    events_stream_df
    .withWatermark("event_time", "10 minutes")   # tolerate up to 10 min late arrivals
)

# Tumbling window aggregation with watermark
result = (
    events_with_watermark
    .groupBy(
        F.window("event_time", "5 minutes"),     # 5-minute tumbling windows
        "customer_id"
    )
    .agg(F.count("*").alias("event_count"))
)
```

**Exam key facts:**

- Watermark **must be set** on the event-time column **before** window aggregation for late data handling.
- `append` output mode is only supported with windowed aggregations when a watermark is set.
- Without a watermark, windowed aggregations must use `complete` or `update` mode.

**2. `foreachBatch` pattern:**

```python
def process_micro_batch(batch_df, batch_id):
    """
    Runs arbitrary batch logic on each micro-batch.
    Enables: multiple table writes, upserts, conditional logic.
    """
    # Upsert micro-batch into a Delta table
    from delta.tables import DeltaTable
    target = DeltaTable.forName(spark, "silver.events")
    (
        target.alias("t")
        .merge(batch_df.alias("s"), "t.event_id = s.event_id")
        .whenMatchedUpdateAll()
        .whenNotMatchedInsertAll()
        .execute()
    )
    # Also write aggregated summary to a second table
    batch_df.groupBy("region").agg(F.count("*").alias("cnt")) \
            .write.format("delta").mode("append").saveAsTable("gold.event_counts")

(
    stream_df.writeStream
    .foreachBatch(process_micro_batch)
    .option("checkpointLocation", "/checkpoints/events/")
    .trigger(processingTime="1 minute")
    .start()
)
```

**3. SCD Type 2 with Delta Lake:**

```python
from delta.tables import DeltaTable
from pyspark.sql import functions as F

target = DeltaTable.forName(spark, "silver.customers_scd2")

(
    target.alias("t")
    .merge(
        source=updates_df.alias("s"),
        condition="t.customer_id = s.customer_id AND t.is_current = true"
    )
    # Close the existing current record when an update arrives
    .whenMatchedUpdate(
        condition="t.email != s.email OR t.plan != s.plan",
        set={
            "is_current":  "false",
            "end_date":    "current_date()",
        }
    )
    .execute()
)

# Insert new current records for changed customers
new_records = (
    updates_df.alias("s")
    .join(
        target.toDF().filter("is_current = true").alias("t"),
        on="customer_id", how="left_anti"   # only customers not in current = true already
    )
    .union(
        updates_df.alias("s")
        .join(
            target.toDF().filter("is_current = true").alias("t"),
            on=(F.col("s.customer_id") == F.col("t.customer_id")) &
               ((F.col("s.email") != F.col("t.email")) | (F.col("s.plan") != F.col("t.plan")))
        )
        .select("s.*")
    )
    .withColumn("is_current",  F.lit(True))
    .withColumn("start_date",  F.current_date())
    .withColumn("end_date",    F.lit(None).cast("date"))
)

new_records.write.format("delta").mode("append").saveAsTable("silver.customers_scd2")
```

### 4.4 Sample Questions and Worked Answers

**Q1:** A streaming query uses `foreachBatch` to MERGE each micro-batch into a Delta table. The job runs for weeks and the Delta table accumulates thousands of small files. Which two steps should be included in a weekly maintenance job?

- A. `OPTIMIZE` and `VACUUM`
- B. `REFRESH TABLE` and `ANALYZE TABLE`
- C. `FSCK REPAIR TABLE` and `REFRESH`
- D. Dropping and recreating the table

**Answer: A** — `OPTIMIZE` compacts small files into larger ones; `VACUUM` removes old files no longer referenced by the transaction log. Both are essential maintenance operations for Delta tables that receive frequent small writes.

---

**Q2:** A watermark of 5 minutes is set on a streaming query using `append` output mode. An event arrives 8 minutes after the end of its window. What happens to that event?

- A. It is included in the next window
- B. It is dropped silently
- C. It is written to a dead-letter table
- D. The pipeline throws a `LateDataException`

**Answer: B** — With `append` mode and a 5-minute watermark, events arriving more than 5 minutes after their window closes are silently dropped. No exception is thrown; no dead-letter sink exists unless you implement one explicitly with `foreachBatch`.

### 7.5 Exam-Day Tips

| Tip                                                    | Detail                                                                                          |
| ------------------------------------------------------ | ----------------------------------------------------------------------------------------------- |
| **Read every answer option** before selecting          | Databricks questions often include two plausible-looking options; the detail matters            |
| **Eliminate obviously wrong answers first**            | Usually 2 of 4 options are clearly wrong; focus on the final 2                                  |
| **Flag and return** on uncertain questions             | Don't spend > 3 min on one question; flag and revisit at the end                                |
| **Watch for "all of the above" / "none of the above"** | These are often traps — verify every option before accepting them                               |
| **Python syntax precision matters**                    | `df.filter()` vs `df.where()` are identical; `.option()` keys are case-sensitive                |
| **Know default values**                                | Default shuffle partitions (200), default broadcast threshold (10 MB), default watermark (none) |
| **Distinguish trigger types**                          | `processingTime`, `once=True` (deprecated), `availableNow=True`, `continuous`                   |
| **Read the scenario carefully**                        | "Most cost-efficient" and "most correct" are different; match the constraint                    |
| **Do not overthink**                                   | If your first instinct based on hands-on experience is A, it is usually right                   |
| **Internet and notes are not permitted**               | Practice with all docs closed to simulate exam conditions                                       |
